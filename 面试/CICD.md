## 💬 面试官：请描述一次你解决过的、具有挑战性的工程化/流水线问题

### 1. 问题背景与现象

“在维护一个大型前端项目时，我们需要在同一个流水线节点内**并行打包**两个版本（根目录版本和子目录 `back/` 版本）。在并发构建时，频繁出现随机的 `Module not found`（特别是针对 `jekins/` 等依赖）以及 `Another instance of yarn is running` 的报错。”

### 2. 问题分析（核心点）

“经过排查，我定位到问题的本质是**全局资源的锁竞争（Race Condition）**：

- **物理隔离不足：** 两个并行的 `yarn install` 默认共享宿主机的全局缓存（`~/.cache/yarn`）。
- **安装与打包的竞态：** 当 A 项目正在打包读取 `node_modules` 时，B 项目的安装进程可能正在修改或重置共享缓存，导致文件句柄失效或目录瞬时空缺，从而引发 Webpack 报错。”

### 3. 解决方案（专家级路径）

“为了彻底根治这个问题，我实施了以下方案：

- **环境隔离（Environment Isolation）：** 在 Shell 脚本中通过 `export YARN_CACHE_FOLDER` 为不同目录指定了唯一且位于工作空间（`$WORKSPACE`）内的物理缓存路径。这避免了全局锁竞争，并确保了缓存随流水线结束而自动清理。
- **环境变量继承确保：** 确保使用 `export` 关键字，使变量能够跨进程传递给 Yarn 的子进程。
- **路径动态化：** 使用 `$(basename $(pwd))` 动态命名缓存目录，提高了脚本的复用性，避免了手动维护路径名的繁琐。”

### 4. 最终结果

“实施隔离策略后，流水线构建的**稳定性提升到 100%**，并发打包时的灵异报错彻底消失。同时，由于缓存依然保留在工作空间内，构建速度并没有因为隔离而受到影响，反而因为减少了排队锁等待而有所提升。”

## 🔍 面试官可能会延伸的问题

如果你说出了上面的方案，资深面试官可能会通过以下问题来深挖你的技术深度：

### Q1：为什么要放 `$WORKSPACE` 目录下，而不是放 `/tmp` 或者固定一个 `/var/yarn-cache`？

- **回答逻辑：** 考查对 **CI/CD 环境清理**的理解。
- **核心：** “为了**生命周期同步**。如果放在 `/tmp` 或全局路径，随着任务增多，磁盘会被撑爆，且存在‘脏缓存’污染下一个任务的风险。放在 `$WORKSPACE` 目录下，Jenkins 会在任务结束或开始前自动执行清理，保证了环境的幂等性。”

### Q2：既然用了 Yarn，为什么不直接用 Yarn Workspaces（工作区）来管理这两个目录？

- **回答逻辑：** 考查对 **Monorepo 架构**的理解。
- **核心：** “这是一个更优的长期架构方案。Workspaces 可以通过一个 `lock` 文件管理所有依赖，只执行一次安装即可。但我之前的方案是针对‘紧急修复’和‘现有物理目录结构’的最快解法。在后续优化中，我们确实可以考虑重构为 Monorepo 来进一步提升依赖一致性。”

### Q3：如果设置了 `YARN_CACHE_FOLDER` 依然报 `Module not found`，你还会从哪些维度排查？

- **回答逻辑：** 考查排查问题的广度（内存、IO、Webpack 配置）。
- **核心：** 1. **内存压力：** 并发构建是否触发了系统的 OOM Killer？ 2. **Webpack 缓存：** Webpack 5 的磁盘缓存（`filesystem cache`）是否也存在目录冲突？ 3. **Node 版本差异：** `hostname` 验证过环境一致，那 Node 版本或用户权限是否存在细微差别导致软链接（Symlink）失效？

### Q4：设置环境变量时，`export` 和在命令前直接写变量名（Inline）有什么区别？

- **回答逻辑：** 考查 **Shell 基础**。
- **核心：** “`export` 是进程级的，影响后续所有命令；Inline 赋值是命令级的，只影响紧随其后的一行。在流水线中，如果后续的 `npm run build` 也要根据环境变量做逻辑判断，`export` 更加稳健。”
